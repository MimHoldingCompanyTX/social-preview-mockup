### Client Portal Mobile Layout & Safari Back Button Fix (2026-03-09 20:15 - 22:00 CDT)
- **User Report:** User reported that the mobile error was fixed in Safari but persisted in Chrome (cache issue). Regression identified: navigating into subfolders was "skipping" the folder level when going back. Later reported that back button from subfolder did nothing on first press and exited portal on second press.

#### Root Cause Analysis:
- `navigateToFolder` was missing `pushNavState` calls (fixed in intermediate turn).
- `handlePopState` logic previously restored current state instead of previous (fixed in intermediate turn).
- **Most Critical Bug (Fixed 22:00):** When restoring state via browser back button, the folder contents (`stepContents`) were not being refreshed. The UI state changed internally but because the files list wasn't re-fetched, the user saw no visual change on first press. This led to second press which exited the portal.

#### Final Resolution & Fixes:
1. **Critical Navigation Logic Fix**: Rewrote `handlePopState` to correctly restore the *previous* state (`navStack[n-2]`) instead of the *current* state.
2. **Dynamic Content Reload**: Modified `handlePopState` to explicitly fetch and reload the appropriate folder contents (step or subfolder) whenever a navigation event is detected.
3. **Gallery State Synchronization**: Added logic to update `currentGalleryFolderId`, `galleryItems`, and `currentGalleryIndex` during state restoration to ensure the linear gallery matches the current folder.
4. **Initial Portal State**: Added `useEffect` to push initial authenticated portal state on mount, ensuring browser back from client list navigates to previous page (password/homepage).
5. **Unified UI & Browser Back Navigation**: Updated `handleBackButton` to use `window.history.back()`.
6. **Model Switch**: Switched back to `openrouter/deepseek/deepseek-v3.2` as requested for coding tasks.
